The Google Cloud FAST Framework

A visual guide to the deployment stages and service modules.

Stage 0: Bootstrap

Foundation Setup

Establishes critical IAM and organization structure, the bedrock for all subsequent stages.

Key Decisions:

  • Organization domain
  • Billing account structure
  • Folder structure
  • Break-glass strategy
  • Automation permissions

Stage 1: ResMan

Core Resources

Deploys foundational projects for centralized logging, security, and networking resources.

Key Decisions:

  • Project naming
  • Core resource regions
  • Log export destinations
  • Security Center setup
  • API enablement

Stage 2: Networking

Connectivity Hub

Configures the network topology, including VPCs, subnets, firewall rules, and hybrid connectivity.

Key Decisions:

  • VPC architecture
  • IP address ranges (CIDRs)
  • Firewall policies
  • DNS strategy
  • VPC Service Controls

Stage 3: Projects

Landing Zones

Creates "landing zones" or tenant projects where applications and workloads will actually run.

Key Decisions:

  • Environment types
  • Default IAM roles
  • Budget alerts
  • Network attachment

Where Can You Start Experimenting?

Once the foundational stages are complete, developers have two paths for getting a project, each suited for different needs.

Available after Stage 0

↓

Manual Project

Ideal for quick, isolated experiments and proofs-of-concept.

  • Speed: Instant, self-service creation.
  • Integration: Standalone, not connected to central networks or policies.
  • Use Case: Testing a new API, learning a service, personal sandbox.

Available after Stage 3

↓

Project Factory

Required for official development, staging, and production workloads.

  • Speed: Automated, may require an approval workflow.
  • Integration: Fully integrated with Shared VPC, logging, and security policies.
  • Use Case: Building and deploying official applications.

Key Learnings in Adopting GCP

a) Networking

GCP: Global VPC

A VPC is a global resource. You can have subnets in different regions (e.g., us-east1, europe-west1) all within the same VPC, simplifying cross-region communication.

AWS: Regional VPC

A VPC is tied to a single region. To connect VPCs in different regions, you must explicitly set up VPC Peering or a Transit Gateway.

b) Organization Structure

GCP: Strict Hierarchy

Follows a rigid structure: Organization > Folders > Projects. This provides clear, top-down inheritance for policies and permissions.

AWS: Flexible Grouping

Uses AWS Organizations to group accounts into Organizational Units (OUs). It's more flexible but requires careful management of Service Control Policies (SCPs).

c) Identity and Access Management (IAM)

GCP: Resource-Centric

Focuses on "who can do what on which resource." Permissions (roles) are granted on specific resources and inherited down the hierarchy.

AWS: Identity-Centric

Focuses on attaching detailed JSON policies to identities (users, groups, roles) that explicitly define what actions they are allowed or denied.

d) Firewalls

GCP: Hierarchical & VPC Firewalls

Uses stateful VPC Firewall Rules applied at the network level. Also offers Hierarchical Firewall Policies that can be enforced from the Organization or Folder level for consistent, top-down security.

AWS: Security Groups & NACLs

Uses a layered approach: stateful Security Groups act as instance-level firewalls, while stateless Network Access Control Lists (NACLs) operate at the subnet level.

e) Service-to-Service Authorization

GCP: Service Accounts are Identities

A Service Account is a distinct, non-human identity. You grant IAM roles to this identity, and a resource (like a VM) runs *as* that identity to inherit its permissions.

AWS: Roles are Assumed

An IAM Role is a set of permissions, not an identity itself. A resource (like an EC2 instance) is given permission to *assume* a role and receive temporary credentials.
